# 기여 가이드

이 문서는 `opengem` 저장소에 참여할 때 따라야 할 개발/PR 가이드입니다.

## 1. 개발 환경

- Node.js: `node` 최신 LTS 권장
- Rust: Tauri 2.x 빌드 대상이므로 `rustup`으로 관리하는 최신 안정 버전 권장
- 패키지 매니저: `pnpm`
- 설치:
  - `pnpm install`

## 2. 실행 및 검증

아래 순서로 동작을 확인합니다.

- `pnpm run env:check`
- `pnpm run dev`
- `pnpm run tauri:dev` (데스크톱 앱 동작 확인)
- 프론트엔드 수정 후: `pnpm run build`

문제가 있으면 로그의 경고/에러부터 먼저 정리하고, 최소 재현 경로를 함께 정리해 커밋 메시지에 포함합니다.

## 3. 브랜치/커밋 규칙

- 기능 단위로 커밋을 나눕니다.
- 메시지는 `feat:`, `fix:`, `chore:` 등 타입으로 시작합니다.
- PR 단위는 하나의 주제만 다루고, 여러 이슈를 묶지 않습니다.

예시:

- `feat: chat panel 스크롤 오프셋 보정`
- `fix: 설정 저장 실패 시 에러 처리 강화`

## 4. 코드 작성 규칙

- TypeScript: 2칸 들여쓰기, 세미콜론 사용
- React는 함수형 컴포넌트 우선
- 의미 있는 식별자 사용(짧은 약어 지양)
- 기존 패턴(스타일/컴포넌트 배치/상태 관리)을 우선 반영
- 불필요한 주석은 피하고, 필요한 경우 의도가 분명한 경우에만 작성

## 5. PR 작성 가이드

PR 생성 전 체크리스트:

- 변경 범위 요약(무엇을 바꿨는지)
- 동작 확인 명령과 결과(예: `pnpm run build` 통과)
- 스크린샷 또는 재현 동영상(UI 변경 시)
- 관련 이슈 번호 연결(있다면)
- 마이그레이션/권한/환경 변수 변경 여부

PR 제목 예시: `feat: 메시지 카드 렌더링 성능 개선`

커밋/PR 제출 전 최소 점검 3단계:

- `pnpm run env:check` 통과 여부를 확인한다.
- 빌드(`pnpm run build`) 또는 실행 흐름(`pnpm run dev`, `pnpm run tauri:dev`)을 검증한다.
- 변경 파일의 의도와 부작용이 문서(README/관련 가이드) 또는 PR 본문에 반영되었는지 확인한다.

PR 본문 템플릿:

```text
## 변경 내용
-

## 검증
- pnpm run env:check
- pnpm run build

## 체크
- [ ] 기존 동작 회귀 없음
- [ ] 코드 포맷/린트 준수
- [ ] 관련 문서 갱신 완료
```

## 6. 이슈 템플릿 사용

버그/개선 이슈는 아래 항목으로 재현 정보를 최소화해 올립니다.

- 발견 환경(OS/브라우저/앱 버전)
- 재현 단계
- 기대 동작/실제 동작
- 관련 로그나 스크린샷

---

이 가이드는 변경될 수 있으며, 중요 변경은 PR 본문에 반영해 팀과 합의합니다.
